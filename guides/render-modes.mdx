`renderMode` controls **when** the canvas re-renders. Motion GPU provides three modes suited to different use cases: `'always'` for continuous animation, `'on-demand'` for interactive updates, and `'manual'` for explicit frame control.

## The three modes

### `'always'` (default)

Renders every frame via `requestAnimationFrame`, as long as `autoRender` is `true`. This is the right choice for animations and simulations that update every frame.

```svelte
<FragCanvas {material} renderMode="always" />
```

**Best for:** continuous animations, particle systems, video textures, GPU simulations.

### `'on-demand'`

Renders only when something triggers an invalidation or `advance()` is called. The canvas stays idle between invalidations, conserving GPU and CPU resources.

      const gpu = useMotionGPU();

      function handleInput(e: Event) {
        radius = parseFloat((e.target as HTMLInputElement).value);
        gpu.invalidate();
      }

      useFrame((state) => {
        state.setUniform('uRadius', radius);
      });
    </script>

    <input type="range" min="0.05" max="0.45" step="0.01"

    import { useState } from 'react';
    import { useMotionGPU, useFrame } from '@motion-core/motion-gpu/react';

    export function Controls() {
      const gpu = useMotionGPU();
      const [radius, setRadius] = useState(0.3);

      useFrame((state) => {
        state.setUniform('uRadius', radius);
      });

      return (
        <input
          type="range" min="0.05" max="0.45" step="0.01"
          value={radius}
          onChange={(e) => {
            setRadius(parseFloat(e.target.value));
            gpu.invalidate();
          }}
        />
      );
    }

**Invalidation sources in `'on-demand'` mode:**

- `gpu.invalidate()` or `state.invalidate()` from user code
- `useFrame` tasks with `autoInvalidate: true` (the default)
- `useFrame` tasks with `invalidation: { mode: 'on-change', token }` when the token changes
- Switching to `'on-demand'` mode auto-injects one initial invalidation

**Best for:** interactive UIs, data visualization, shader editors, infrequent updates.

### `'manual'`

Renders only after an explicit `advance()` call. The canvas does not react to invalidations.

**Best for:** frame-by-frame capture, testing, server-side rendering scenarios, offline export.

## `invalidate()` and `advance()`

Both are available on the `FrameState` object inside `useFrame` callbacks and on the `useMotionGPU()` return value.

| Method | Effect |
| --- | --- |
| `invalidate(token?)` | Marks the canvas dirty for `'on-demand'` mode. The optional `token` enables per-source invalidation tracking. |
| `advance()` | Requests a single render in `'manual'` mode. Has no effect in `'always'` or `'on-demand'`. |

## Switching modes at runtime

Use the `renderMode` setter from `useMotionGPU()` to switch modes at runtime:

```typescript
const gpu = useMotionGPU();

gpu.renderMode.set('on-demand');  // Switch to on-demand
gpu.renderMode.set('manual');     // Switch to manual
gpu.renderMode.set('always');     // Back to always
```

## The `autoRender` gate

`autoRender` is a boolean that acts as a hard override on all rendering, regardless of mode. When `false`, no frames are submitted:

```typescript
gpu.autoRender.set(false); // Stop all rendering
gpu.autoRender.set(true);  // Resume
```

You can also set it as a `FragCanvas` prop:

```svelte
<FragCanvas {material} renderMode="always" autoRender={false} />
```

## Performance implications

<CardGroup cols={3}>
  <Card title="'always'" icon="play">
    Highest GPU utilization. `requestAnimationFrame` fires continuously. Ideal when every frame changes.
  </Card>
  <Card title="'on-demand'" icon="bolt">
    GPU is idle between invalidations. Reduces power consumption and heat on battery-powered devices. Requires explicit invalidation from user interactions or state changes.
  </Card>
  <Card title="'manual'" icon="pause">
    Zero background GPU activity. Requires explicit `advance()` for each frame. Use for capture workflows or unit tests.
  </Card>
</CardGroup>

<Tip>
  Start with `'always'` during development, then profile and switch to `'on-demand'` for production if the canvas updates infrequently.
</Tip>
